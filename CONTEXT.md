# Application Context

This is a Laravel 13 + Inertia Vue 3 application using the Vue starter kit as a base. The current app is an internal support ticket tracker for authenticated and verified users.

## Stack

- PHP 8.4, Laravel 13, Fortify, Inertia Laravel 3, Wayfinder.
- Vue 3, Inertia Vue 3, Tailwind CSS 4, Reka UI-based UI components.
- Pest 4 for tests.
- SQLite is the local/default database engine.
- Laravel Herd serves the app locally; do not start an additional dev server unless specifically needed for Vite assets.

## Product Shape

- The app is intentionally KISS: conventional CRUD pages, small controllers, direct Eloquent models, and simple Inertia props.
- It is for one authenticated internal user managing external customer issues.
- There are no roles, admin screens, customer portal, email sending, attachments, SLA workflows, or reminders.
- Tickets are hard-deleted. Related messages and pivot tag links are deleted by database cascade; reusable tag records remain.

## Authentication

- Routes under `auth` and `verified` are the protected application area.
- Fortify powers authentication.
- Users only see their own tickets through `Ticket::scopeOwnedBy()` and policies.

## Main Routes

- `/` renders the `Welcome` Inertia page.
- `/dashboard` renders ticket stats through `DashboardController`.
- `/tickets` is a resource controller handled by `TicketController`.
- `/tickets/{ticket}/messages` stores ticket messages.
- `/tickets/{ticket}/messages/{message}` updates and deletes scoped messages.

## Ticket Domain

- `Ticket` belongs to `User`.
- `Ticket` has many `TicketMessage` records.
- `Ticket` belongs to many reusable `Tag` records.
- `Tag` uses a case-insensitive slug generated by `Tag::slugFor()`.
- Ticket numbers are generated after creation as `TCK-000001` style values.

## Enums

- Status: `open`, `pending`, `closed`.
- Priority: `low`, `normal`, `high`, `urgent`.
- Department: `support`, `billing`, `technical`, `sales`.
- Sentiment: `positive`, `neutral`, `negative`.
- Message type: `customer_message`, `agent_reply`, `internal_note`.

## Ticket Workflows

- Creating a ticket creates the ticket, syncs tags, and creates the initial `customer_message`.
- New tickets start as `open` and set `last_message_at` to creation time.
- Updating a ticket can change status, priority, department, sentiment, customer details, and tags.
- Setting status to `closed` sets `closed_at`; moving away from `closed` clears `closed_at`.
- Adding a `customer_message` reopens the ticket and clears `closed_at`.
- Adding an `agent_reply` sets active tickets to `pending`; closed tickets stay closed unless explicitly reopened.
- Adding any message updates `last_message_at`.
- Deleting a message is allowed, but the final remaining message cannot be deleted.

## Frontend Structure

- Inertia pages live in `resources/js/pages`.
- Ticket pages live in `resources/js/pages/tickets`.
- Shared app UI components live in `resources/js/components` and `resources/js/components/ui`.
- Use Wayfinder-generated imports from `@/actions` and `@/routes` instead of hardcoded URLs.
- `FormSelect.vue` wraps the UI select component for Inertia `<Form>` submissions using a hidden input.
- `FormSelect.vue` supports `placeholderValue` so UI placeholder options can submit an empty string without violating Reka UI's non-empty `SelectItem` value requirement.
- `TagInput.vue` is the chip-style reusable tag input.
- `Input.vue` supports `modelValue` when controlled and `defaultValue` for normal Inertia form usage.

## Backend Structure

- Controllers:
  - `DashboardController`
  - `TicketController`
  - `TicketMessageController`
- Form requests:
  - `StoreTicketRequest`
  - `UpdateTicketRequest`
  - `StoreTicketMessageRequest`
  - `UpdateTicketMessageRequest`
- Models:
  - `Ticket`
  - `TicketMessage`
  - `Tag`
- Policies protect ticket and message actions.

## Seed Data

- `DatabaseSeeder` creates the main user and calls `TicketSeeder`.
- `TicketSeeder` creates up to 100 sample tickets with messages and reusable tags.

## Tests

- `tests/Feature/TicketTest.php` covers ticket CRUD, authorization, and deletion behavior.
- `tests/Feature/TicketMessageTest.php` covers message creation, updates, deletion rules, and status transitions.
- `tests/Feature/DatabaseSeederTest.php` verifies seeded ticket data.

## Verification Commands

- PHP formatting: `vendor/bin/pint --dirty --format agent`
- PHP tests: `php artisan test --compact`
- Frontend types: `npm run types:check`
- Frontend lint: `npm run lint:check`
- Frontend build: `npm run build`

## Known Notes

- `npm run build` may show non-fatal Rolldown `INVALID_ANNOTATION` warnings from `reka-ui` / nested `@vueuse/core`; the build still succeeds.
- Follow `AGENTS.md` and Laravel Boost guidance for future changes.
